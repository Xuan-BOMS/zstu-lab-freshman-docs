# 实验室新人 STM32 学习路径

## 1. 前言

这篇学习路径主要写给实验室新人和大一新生，目标不是让你在最短时间内把所有 STM32 外设都学一遍，而是给你一条**能走通、能做项目、能参加比赛、能继续进阶**的主线。

网上关于 STM32 的资料很多，但对新手来说，最大的问题往往不是“找不到资料”，而是“资料太多，不知道该按什么顺序学”。这篇文章希望解决的就是这个问题。

本文会以 `STM32 + HAL + CubeMX + 现代开发环境` 为主线，尽量让你从一开始就走一条更贴近现在开发方式的路线，而不是把大量时间花在历史包袱上。

---

## 2. 学完这条路线后，你应该达到什么水平

如果你能比较认真地按这条路线学习 3 个月到半年，正常应该具备以下能力：

- 能独立完成一个基础 STM32 小项目
- 能参与电赛 / 工创赛中的嵌入式部分
- 能做基础物联网相关毕设或实验室项目
- 能读懂中等复杂度 STM32 开源工程
- 能继续迁移到 GD32、RA 等其他 MCU 平台

这里的“学会”不是指你会点亮 LED，也不是指你能把例程跑起来，而是指你开始具备：

- 基本工程搭建能力
- 基本调试能力
- 基本驱动编写和模块抽象能力
- 基本查手册解决问题的能力

---

## 3. 为什么以 STM32 为主线

STM32 依然是实验室新人最合适的主线平台，原因很简单：

- 资料多，生态成熟
- 学校项目、比赛、开源工程里都很常见
- 工具链完善，配套外设和模块丰富
- 入门后迁移到其他 MCU 成本较低

本文虽然以 STM32 为主线，但并不是说以后只能学 STM32。相反，STM32 更像是你进入嵌入式开发世界的一块跳板。等你把 STM32 的开发思路、调试方式、工程组织方式学明白之后，再去看 GD32、RA，甚至 Zephyr 生态下的其他平台，都会容易很多。

---

## 4. 为什么本文不建议从老路线入门

很多 STM32 老教程都会从 `STM32F103 + 标准库` 开始讲，这条路线在过去确实很常见，也积累了大量中文资料。但我并不建议实验室新人把它作为主线，主要原因有两个：

- 老教程虽然多，但路径已经过时
- 很容易把注意力放在历史包袱上，而不是现代开发方法上

新生如果从老路线入门，很容易出现以下情况：

- 花很多时间记标准库接口
- 工程搭建方式比较陈旧
- 对 CubeMX / HAL / 新系列芯片不熟悉
- 后续面对新芯片和新工具链时反而要重新适应

这并不是说老资料完全没用。老资料在以下场景仍然有价值：

- 理解 STM32 的一些基础思想
- 查找外设使用案例
- 看老项目时快速补背景

但对于实验室新人来说，主线更适合直接从**新系列 STM32 + HAL + CubeMX** 开始。

---

## 5. STM32 芯片系列怎么认识

STM32 芯片系列很多，新人不需要一开始就背全，但至少要对几个大方向有概念。

### 5.1 老系列

常见老系列包括：

- `F1`
- `F4`
- `F7`

这些系列资料多、案例多、历史项目多，所以你在网上和实验室旧工程里仍然会经常见到它们。

### 5.2 新系列

常见新系列包括：

- `C`
- `G`
- `H`
- `U`

这些系列更符合现在的开发链路，官方工具支持也更贴近当前生态。

### 5.3 本文为什么主推新系列

对于新人来说，新系列更适合作为主线，原因包括：

- 更适合直接学习 `CubeMX + HAL`
- 工程搭建和外设配置更自然
- 某些复杂外设在 HAL 下更容易配置和管理
- 更符合后续比赛和项目开发的实际需求

## 6. 本文唯一主推的入门系列：STM32G474

本文唯一主推的入门系列是 `STM32G474`。

原因并不是“它最便宜”或者“所有人都必须买它”，而是因为它作为入门主线非常均衡：

- 外设丰富
- 性能足够
- 模拟外设能力比较强
- 更适合比赛和项目中的采样、控制、信号处理类任务
- 更适合直接进入 `HAL + CubeMX` 的现代开发路线

如果你后面要做一些和以下方向相关的内容，`G4` 会更有优势：

- ADC 采样
- DAC 输出
- OPAMP 相关应用
- 信号采样与处理
- 比赛中的控制类题目

G4作为ST新的General系列，各方面性能非常均衡，给到了丰富的模拟外设，足够大小的RAM和FLASH，以及170MHz的主频。实际性能可以参考[CoreMark](https://www.eembc.org/coremark/index.php)，STM32G4的CoreMark成绩在170MHz下达到了581.5分，CoreMark/MHz成绩为3.42；作为对比常见的C8T6分数为159.15，CoreMark/MHz成绩为2.21。

不过需要注意的是，这个成绩和使用的编译链，代码所在的RAM等存在关系，比如G4可以**使用ART代码加速并将代码放置在CCMSRAM中**以达到最高性能。参考[ST论坛](https://shequ.stmicroelectronics.cn/thread-632287-1-1.html)。这些针对芯片的深入优化可以通过查看芯片的高级外设进行学习。

另外G4带有单精度浮点运算单元，可以加快浮点运算速度，需要在CubeMX生成时勾选，或者在Keil、IAR工程中启用。

### 6.1 一个核心板示例

我自己购买的核心板示例：

[STM32G474 核心板示例](https://e.tb.cn/h.ijbaG8VUEqDAn9L?tk=Qc2M5Xk2QIw)

板子本身不是重点，重点是你要尽量围绕**新系列 + 核心板 + 现代工具链**来搭建自己的起步环境。

## 7. 开发板怎么选

对于实验室新人来说，没有必要一上来就买功能特别多的大开发板。大而全的板子看起来很强，但对新手来说往往会带来几个问题：

- 成本更高
- 不知道哪些部分是真正需要学的
- 很容易变成“照着例程点功能”

更推荐的方式是：

- 没有特殊需求时优先买核心板
- 优先跟随实验室已有芯片路线
- 保证后续资料和项目环境统一

这样做的好处是：

- 你会更清楚“最小系统”是怎么工作的
- 更容易理解电源、时钟、下载、串口这些基础部分
- 更容易形成自己的硬件和软件抽象能力

## 8. 下载器 / 调试器怎么选

下载器和调试器并不是“能烧进去就行”，不同工具在调试体验上差异很大。

### 8.1 ST-Link V2 / V3

`ST-Link` 是 STM32 生态里最常见的下载调试工具。

特点：

- 官方生态适配好
- 使用门槛低
- 资料多
- 对 STM32 支持自然

两者的区别是：ST-Link V2 是最常见的版本，价格便宜但比较老，速度较慢；ST-Link V3 可以在淘宝购买，价格大约 80 元，速度更快，性能更好。不过ST-Link V3需要自己焊接2.0mm的牛角座，并需要购买2.0转2.54的杜邦线来配套使用。

### 8.2 DAPLink

我个人比较常用的是无线 `DAPLink`：[我购买的无线 DAPLink](https://e.tb.cn/h.ijozza1rBT2uEYQ?tk=NxrY5Xkfmyb)

实验室也有正点原子的无线 DAPLink。

它的优势主要在于：

- 使用方便
- 无线场景下体验好
- 对于日常下载调试已经足够

### 8.3 J-Link

`J-Link` 的优势主要在于调试体验，尤其是：

- RTT
- 更成熟的调试工具链
- 更强的一些调试扩展能力

但对新人来说，如果你暂时不用 `RTT`，其实并不一定非要上 `J-Link`。

另外 `J-Link` 的烧录速度也非常快，它有特有的烧录算法，可以实现增量烧录。

### 8.4 SWO 和 RTT

很多新人调试时只会用串口打印（`printf` 重定向到 UART），这当然能用，但有两个明显的缺点：

- 占用一个串口外设和对应引脚
- 波特率有限，打印太多会明显拖慢程序运行

`SWO` 和 `RTT` 是两种不依赖串口的调试输出方式，都通过调试器通道传输数据，不占用额外引脚。

#### SWO（Serial Wire Output）

- 利用 SWD 接口上的 `SWO` 引脚输出调试信息
- 需要调试器支持 SWO（ST-Link V2/V3、J-Link 都支持）
- 需要在 MCU 端配置 ITM（Instrumentation Trace Macrocell）
- 速度比串口快，但仍然是单向输出（MCU -> PC），不能从 PC 往回发数据
- 适合替代 `printf` 调试，在 IDE 的 SWO Viewer 窗口中查看输出

ITM等内容可以看《Cortex-M4 开发权威指南》，里面有详细的介绍。

#### RTT（Real Time Transfer）

- SEGGER 提供的调试输出方案，通过 J-Link 的调试通道双向传输数据
- 不需要额外引脚，不需要额外外设，甚至不需要额外中断
- 原理是在 RAM 中开辟一块缓冲区，J-Link 通过调试接口直接读写这块内存
- 速度非常快，对程序运行时序的影响极小
- 支持双向通信（MCU <-> PC）
- 需要 J-Link 调试器，使用 SEGGER 提供的 RTT Viewer 查看输出

#### 怎么选

| | 串口打印 | SWO | RTT |
|---|---|---|---|
| 需要额外引脚 | 是 | 是（SWO 引脚） | 否 |
| 需要额外外设 | 是（UART） | 否 | 否 |
| 方向 | 双向 | 单向（MCU->PC） | 双向 |
| 速度 | 受波特率限制 | 较快 | 很快 |
| 对时序影响 | 较大 | 较小 | 极小 |
| 调试器要求 | 无 | ST-Link / J-Link | J-Link |

对于新人来说，串口打印入门最简单，先用着没问题。但当你发现打印太多导致程序变慢、或者串口已经被其他模块占用时，就应该考虑换到 SWO 或 RTT。尤其是做对时序敏感的调试（比如电机控制、高速采样），RTT 几乎是必需的。

以上提到的一些限制比如 SWO 无法接收或RTT必须使用 J-Link 等都是可以通过其他方式解除限制的，不过不是稳定的方案，感兴趣的可以自己学习。

### 8.5 示波器和逻辑分析仪

新人一般不需要自己购买，优先使用实验室现有设备即可。

## 9. 推荐开发环境

本文推荐的主线开发环境不是传统的纯 `Keil` 路线，而是更偏现代开发体验的方案。

### 9.1 主推荐路线

- `CubeMX + HAL`
- `IAR + VS Code`

### 9.2 补充路线

- `CubeMX + Keil`

这条路线资料非常多，所以即使你不用它做主力环境，后续查资料时也一定会遇到。

### 9.3 扩展路线

- `CMake + CLion`
- `Zephyr + west`

其中 `Zephyr` 的一个入门参考链接：

[Zephyr RTOS 入门](https://www.ruanx.net/zephyr-rtos/)

Zephyr是一个更加现代的嵌入式RTOS，现代的点包括和Linux类似的思想，使用DTS和Board来控制硬件可以让你快速的在项目中切换不同的主控；同时Zephyr有完善的跨平台开发工具链和医疗设备认证，并且拥有自己的BLE、Wifi等无线协议栈。

### 9.4 为什么推荐 IAR + VS Code

这是我目前觉得在 VS Code 中体验最好、功能也比较完整的一套开发环境。原因包括：

- VS Code 的界面和交互更现代
- 跳转、搜索、查看符号、阅读工程体验更好
- IAR 的工具链本身比较专业，功能不缺
- 二者结合后比较均衡

### 9.5 这套环境至少要覆盖哪些日常动作

- 如何创建工程

​	CubeMX 可以直接生成 IAR 工程，IAR 工程是 EWARM 文件夹下的 .eww 工程文件。双击可以直接在 IAR 中打开。

- 如何导入或生成工程

  直接打开 .eww 文件就可以直接打开 IAR 工程。在 CubeMX 路径或在 EWARM 路径打开 vscode，vscode 中的 IAR 插件就可以直接解析 IAR 工程。

  IAR 界面与 Keil 类似，左侧为工程文件目录，右键文件夹可以 add 文件，右键工程可以设置编译链（IAR 或 GNU 扩展）、语言等，包括添加 Inc 路径、切换不同类型的下载器等操作都在这个总的 Options 中。注意工程是**最顶层的工程文件**，其他的文件夹都在工程下。

  vscode 中添加文件可能无法同步到 IAR 工程中，无论是添加文件还是修改工程配置，都建议在 IAR 中修改，修改完成后在 vscode 中重新打开就会重新解析 .eww 文件。

- 如何编译

  在 IAR 中进行编译的快捷键是 **F7**，位置在顶部工具栏；vscode 中的按键是**编译并下载**，在 vscode 中的 IAR 插件的顶端 TOOLBAR 中的左上角。

- 如何跳转和查找符号

​	在 vscode中直接使用 F12 进行跳转即可。

- 如何烧录

​	在 vscode 中点击编译并下载即可；在 IAR 中快捷键为 **Ctrl + D**，位置在编译按键的右端。

- 如何调试

  在 IAR 中的调试方法如上；在 vscode 中切换到运行和调试（Ctrl + Shift + D）后，在顶端将调试配置改为 IAR C-SPY Debug 后，再根据自己的需要选择 attach 或 launch。

- 如何查看变量和调用栈

  在 vscode 中打断点可以直接查看调用堆栈，调用堆栈就是函数的调用链，**如果出现硬件错误，可以通过查看调用堆栈来尝试分析系统在哪里出现了崩溃，这是一个非常有用的调试工具**。

  另外变量也可以在 vscode 的调试界面进行查看。IAR 插件还会解析 MCU 厂商提供的 xml 文件，即你可以在 vscode 中直接通过查询外设名字的方式来查看对应外设的寄存器的值，对应 MCU 的数据手册可以排查对应外设的问题。

- 如何定位编译报错和链接报错

  这部分建议使用 AI，不同工程可能会出现不同的问题，AI 能帮我们解决大部分的问题。

## 10. 学 STM32 前需要什么基础

新生不需要把所有基础课都学完才能开始学 STM32，但至少要具备下面这些最低基础：

### 10.1 C 语言

至少要会：

- 基本语法
- 函数
- 指针
- 结构体
- 数组
- 基本的模块拆分

如果这些内容还不熟，后面写驱动和看工程会非常吃力。

### 10.2 基础电路知识

至少要知道：

- 电平输入输出的基本概念
- 上拉下拉
- 电源和地
- 模拟信号和数字信号的区别

### 10.3 原理图阅读能力

拿到一块开发板或核心板，第一件事就是看原理图。你至少要能做到：

- 找到 MCU 引脚和板上外设/接口的对应关系
- 看懂电源部分的基本拓扑
- 看懂串口、I2C、SPI 等接口在原理图上的连接方式
- 根据原理图判断一个引脚应该配置成什么模式

这个能力越早建立越好，否则后面接模块、查问题都会很被动。

### 10.4 Git 版本控制

在实验室里协作和参加比赛，Git 几乎是必备技能。你至少要会：

- `git init` / `git clone`
- `git add` / `git commit`
- `git push` / `git pull`
- 分支的基本概念和使用
- 处理简单的合并冲突

不需要一开始就精通，但从第一个项目开始就用 Git 管理代码是一个好习惯。推荐在 GitHub 或 Gitee 上建自己的学习仓库。

除此之外，可以联系社长在实验室内部的 GitLab 中创建账号。

### 10.5 基本外设概念

不要求一开始就会用，但最好知道这些词是什么意思：

- GPIO
- 串口
- 中断
- 定时器
- ADC
- I2C
- SPI

### 10.6 代码规范意识

从一开始就养成基本的代码规范习惯，后面协作和维护会轻松很多：

- 变量和函数命名风格统一（推荐 `snake_case`）
- 文件组织清晰，驱动、业务、配置分开放
- 宏定义用大写加下划线
- 关键逻辑加必要的注释，但不要注释每一行

不需要一开始就追求完美，但至少不要写出"只有自己看得懂"的代码。

## 11. STM32 学习主线顺序

本文建议的主线顺序是：

`GPIO -> UART -> 中断 -> 定时器 / PWM -> ADC/DAC -> I2C -> SPI -> DMA`

> 注意：PWM 本质上是定时器的一种输出模式，不是独立外设，所以放在一起学。

后面每学一个外设，都建议按同样的方式整理：

- 这个外设是干什么的
- 在比赛/项目里常见在哪
- 用 HAL 怎么上手
- 学到什么程度算会了
- 常见坑
- 如何调试
- 建议练习

---

## 12. HAL 为主，但必须理解的底层逻辑

本文虽然以 HAL 为主线，但不代表你只要会点 CubeMX 就够了。

新生至少要理解这些底层逻辑：

- GPIO 的输入输出本质
- 时钟树的基本概念
- 中断响应机制
- 定时器的大致工作模型
- DMA 的核心思想
- HAL 配置和底层寄存器之间的大致对应关系

学习 HAL 的正确方式应该是：

- 用 HAL 提高开发效率
- 用底层理解保证你不会变成“只会点配置的人”

## 13. 哪些外设必须重点掌握

### 13.1 必须熟练的外设

这些外设在比赛和项目里使用频率很高，必须熟练掌握：

- `UART`
- `I2C`
- `SPI`
- `DMA`
- `ADC`
- `DAC`

### 13.2 必须理解基本原理的内容

这些内容是所有后续开发的基础：

- `GPIO`
- 时钟
- 中断
- 定时器 / `PWM`
- 看门狗（`IWDG` / `WWDG`）

### 13.3 建议了解的外设

这些外设在特定场景下很有用，建议在主线学完后了解：

- `CAN` / `FDCAN`：控制类比赛题目和车辆/工业场景常见，G474 本身支持 FDCAN
- `RTC`：物联网项目中需要时间戳时会用到
- 内部 `Flash` 读写：存储校准参数、配置数据等场景很常见
- `USB`：G474 支持 USB，作为进阶方向值得了解

### 13.4 必须学会查手册判断能力边界的内容

对于一些更复杂或更偏模拟的外设，你不一定一开始就要完全掌握，但必须知道如何查手册看指标、看能力、看约束，例如：

- `OPAMP`
- ADC 的精度、速度、采样条件
- 外设能否串联组合使用
- 某个芯片是否适合某个比赛题目

这类能力会直接影响你在方案设计阶段的判断。

## 14. 调试能力才是 STM32 学习的真正门槛

很多同学会发现，STM32 最难的部分并不是“配置出来”，而是“调出来”。

真正拉开差距的，往往是调试能力。

### 14.1 常见问题

- 串口粘包 / 分包
- I2C 无响应
- SPI 模式错误
- DMA 不进中断

### 14.2 为什么会卡在这些问题上

因为这些问题很少能通过“多点几下 CubeMX”解决，它们通常涉及：

- 协议时序
- 缓冲区管理
- 中断处理
- 引脚复用
- 时钟配置
- 外设状态机

### 14.3 调试时应该学会什么

**IDE 内调试（最高频的手段）：**

- 打断点、单步执行
- 查看变量值（Watch 窗口）
- 查看调用栈（Call Stack）
- 查看和修改寄存器
- 查看内存内容

**外部观测：**

- 看串口日志（或 SWO、RTT）
- 看波形（示波器）
- 看逻辑分析结果
- 看中断有没有进
- 看 DMA 有没有搬运
- 看发送和接收双方的时序是否一致

IDE 内调试和外部观测要结合使用。很多时候光看串口日志定位不了问题，必须断点进去看变量状态；反过来，时序问题光靠断点也看不出来，需要示波器或逻辑分析仪。

## 15. 比赛导向的学习方式

我更建议新人采用比赛导向的学习方式，因为它更容易让你理解“为什么要学这些外设”，也更容易建立兴趣。

常见的比赛/项目需求包括：

- 信号采样与处理
- 数据上传
- 人机交互
- 传感器采集

你学每个外设时，都可以问自己一个问题：

> 这个外设在真实项目里是怎么用起来的？

这样学出来的东西，后面更容易拼成系统。

## 16. 推荐练手项目

本文推荐的练手项目顺序如下：

1. `DMA 串口接收框架`
2. `ADC 采样和滤波`
3. `传感器采集节点`
4. `接入 ThingsBoard 的小系统`

### 16.1 DMA 串口接收框架

训练目标：

- 理解串口接收流程
- 处理不定长数据
- 建立 buffer 管理意识
- 为后续接模块打基础

这个项目会自然地用到两个重要的编程模式：

- **环形缓冲区（Ring Buffer）**：管理接收数据的标准方式，几乎所有串口接收框架都会用到
- **状态机**：解析协议帧时的核心思路，比如按"帧头 -> 长度 -> 数据 -> 校验"的状态逐步解析

建议在这个阶段就把这两个模式理解清楚，后面做任何涉及数据收发的项目都用得上。

### 16.2 ADC 采样和滤波

训练目标：

- 理解 ADC 基本采样流程
- 学会做基础滤波
- 理解采样频率、抗干扰、校准这些概念
- 建立一点模拟前端意识

这一部分不建议只停留在“读出一个数”，还应该开始思考：

- 为什么采样值会抖动
- 采样频率怎么选
- 怎么滤波
- 什么是校准
- 模拟信号输入链路需要注意什么

### 16.3 传感器采集节点

建议从这些典型传感器入手：

- `IMU`
- 气压传感器
- 光照传感器

训练目标：

- 学会和常见 I2C / SPI 传感器通信
- 学会初始化和读数流程
- 学会把多个模块组织到同一个工程里

使用IMU是因为在正常使用过程中IMU的采样频率会很高，能够训练相关的数据传输、解析能力。

### 16.4 接入 ThingsBoard 的小系统

训练目标：

- 打通“设备采集 -> 数据发送 -> 平台接收”的基本链路
- 理解一个简单物联网系统是怎么拼起来的
- 为后续毕业设计或实验室项目做准备

## 17. 如何写可维护、可移植的驱动

新人写驱动时最容易出现的问题，是把代码和某一个芯片、某一组引脚、某一种底层实现写死，导致代码虽然能跑，但后面很难维护和迁移。

所以从一开始就应该建立一点抽象思维。

### 17.1 为什么不要强耦合到某个芯片

因为工程一旦变大，你就会遇到这些问题：

- 换一个接口就要重写很多代码
- 换一个芯片就很难复用
- 上层业务逻辑和底层硬件耦合太紧
- 后续读代码的人很难维护

### 17.2 推荐的基本思路

- 用结构体描述设备对象
- 用函数指针描述底层操作
- 分离接口层、硬件层、业务层
- 上层尽量只关心“我要读数据”，而不是“这个芯片具体怎么拉引脚”

### 17.3 一个通用 I2C 传感器驱动抽象模板

下面用一个简化的例子展示怎么把 I2C 传感器驱动做基本的抽象。核心思路是：**上层驱动只关心”读写寄存器”，不关心底层 I2C 具体怎么实现。**

#### 第一层：总线操作接口

把底层的 I2C 读写抽象成函数指针，这样换平台时只需要替换这一层：

```c
/* sensor_i2c.h */

#include <stdint.h>

/**
 * I2C 总线操作接口
 * 移植时只需要实现这两个函数指针
 */
typedef struct {
    /**
     * I2C 写操作
     * @param dev_addr  设备 7 位地址
     * @param reg_addr  寄存器地址
     * @param data      要写入的数据
     * @param len       数据长度
     * @return          0 成功，非 0 失败
     */
    int (*write)(uint8_t dev_addr, uint8_t reg_addr, const uint8_t *data, uint16_t len);

    /**
     * I2C 读操作
     * @param dev_addr  设备 7 位地址
     * @param reg_addr  寄存器地址
     * @param data      读取数据存放的缓冲区
     * @param len       要读取的长度
     * @return          0 成功，非 0 失败
     */
    int (*read)(uint8_t dev_addr, uint8_t reg_addr, uint8_t *data, uint16_t len);
} sensor_i2c_t;
```

#### 第二层：设备对象

每个传感器用一个结构体描述，包含地址和总线接口的指针：

```c
/* sensor_example.h */

#include “sensor_i2c.h”

typedef struct {
    uint8_t             addr;       /* 设备 I2C 地址 */
    const sensor_i2c_t *bus;        /* 指向总线操作接口 */
} sensor_example_t;

/* 初始化：检查设备 WHO_AM_I 寄存器，确认通信正常 */
int sensor_example_init(sensor_example_t *dev);

/* 读取数据（具体含义取决于传感器类型） */
int sensor_example_read(sensor_example_t *dev, float *value);
```

#### 第三层：驱动实现

驱动内部只通过 `dev->bus->read` / `dev->bus->write` 操作总线，完全不涉及具体平台：

```c
/* sensor_example.c */

#include “sensor_example.h”

#define WHO_AM_I_REG    0x0F
#define EXPECTED_ID     0xB1
#define DATA_REG        0x28

int sensor_example_init(sensor_example_t *dev)
{
    uint8_t id = 0;
    if (dev->bus->read(dev->addr, WHO_AM_I_REG, &id, 1) != 0) {
        return -1;  /* 通信失败 */
    }
    if (id != EXPECTED_ID) {
        return -2;  /* 设备 ID 不匹配 */
    }

    /* 这里可以继续写配置寄存器的初始化流程 */
    return 0;
}

int sensor_example_read(sensor_example_t *dev, float *value)
{
    uint8_t raw[2] = {0};
    if (dev->bus->read(dev->addr, DATA_REG, raw, 2) != 0) {
        return -1;
    }

    int16_t raw_value = (int16_t)((raw[1] << 8) | raw[0]);
    *value = raw_value * 0.01f;  /* 假设的换算系数 */
    return 0;
}
```

#### 第四层：平台适配（以 STM32 HAL 为例）

移植到具体平台时，只需要实现 `sensor_i2c_t` 里的两个函数：

```c
/* sensor_i2c_stm32.c */

#include “sensor_i2c.h”
#include “i2c.h”  /* CubeMX 生成的 I2C 头文件 */

static int stm32_i2c_write(uint8_t dev_addr, uint8_t reg_addr,
                           const uint8_t *data, uint16_t len)
{
    return HAL_I2C_Mem_Write(&hi2c1, dev_addr << 1, reg_addr,
                             I2C_MEMADD_SIZE_8BIT,
                             (uint8_t *)data, len, 100) == HAL_OK ? 0 : -1;
}

static int stm32_i2c_read(uint8_t dev_addr, uint8_t reg_addr,
                          uint8_t *data, uint16_t len)
{
    return HAL_I2C_Mem_Read(&hi2c1, dev_addr << 1, reg_addr,
                            I2C_MEMADD_SIZE_8BIT,
                            data, len, 100) == HAL_OK ? 0 : -1;
}

/* 导出这个接口实例，供上层使用 */
const sensor_i2c_t sensor_i2c_stm32 = {
    .write = stm32_i2c_write,
    .read  = stm32_i2c_read,
};
```

#### 实际使用

```c
#include “sensor_example.h”

extern const sensor_i2c_t sensor_i2c_stm32;

sensor_example_t my_sensor = {
    .addr = 0x68,
    .bus  = &sensor_i2c_stm32,
};

void app_init(void)
{
    if (sensor_example_init(&my_sensor) != 0) {
        /* 初始化失败，处理错误 */
    }
}

void app_loop(void)
{
    float value;
    if (sensor_example_read(&my_sensor, &value) == 0) {
        /* 使用 value */
    }
}
```

#### 这样组织的好处

- **换平台**：只需要重新实现 `sensor_i2c_t` 里的两个函数，驱动代码一行不改
- **换传感器**：新传感器只需要新写一个 `sensor_xxx.c`，复用同一套总线接口
- **同一总线挂多个设备**：多个设备对象共用同一个 `sensor_i2c_t` 实例，只是地址不同
- **单元测试**：可以写一个模拟的 `sensor_i2c_t` 在 PC 上测试驱动逻辑

这个模板不是唯一正确的写法，但它展示了最核心的思路：**用函数指针把”做什么”和”怎么做”分开**。你在 GitHub 上看到的大部分质量较好的传感器驱动库（比如 ST 官方的 `stts751-pid`、`lsm6dso-pid` 等），基本都是这个套路。

### 17.4 从 GitHub 学驱动时要看什么

看别人的驱动，不要只盯着“这段代码能不能跑”，更应该看：

- 设备对象怎么设计
- 初始化流程怎么组织
- 收发接口怎么抽象
- 配置项怎么管理
- 上层如何调用底层

学习目标不是“抄代码”，而是“学别人怎么组织代码”。

## 19. 物联网部分怎么入门

物联网部分在本文里的目标不是“深入网络协议栈”，而是：

> 能把模块接起来，能把数据发出去，能把平台接通。

### 19.1 常见方式

- `Wi-Fi`
- `4G`
- `LoRa`
- `BLE`（蓝牙低功耗）

### 19.2 各种方式的特点

`Wi-Fi`：

- 成本低
- 适合有网络环境的场景
- 比较适合实验室内快速验证

`4G`：

- 覆盖广
- 适合独立联网
- 更贴近一些实际部署场景

`LoRa`：

- 传输距离远
- 功耗低
- 但速率低、延迟较大

`BLE`：

- 短距离、低功耗
- 适合和手机直接交互
- 穿戴设备、传感器节点等场景很常见

新人不需要一开始就把这些协议细节全搞懂，但至少要知道：

- 什么时候该选哪一种
- 不同方案的大致优缺点是什么

## 20. ThingsBoard 接入的定位

本文对 ThingsBoard 的要求是：

- 知道它是干什么的
- 能完成最基础的接入
- 理解基本的数据流

接入方式可以提到：

- `HTTP`
- `MQTT`

其中 MQTT 是物联网场景下最常见的协议，你至少要理解这几个核心概念：

- **Broker**：消息中转服务器，设备和平台都连接到它
- **Topic**：消息的"频道"，发布和订阅都基于 topic
- **Publish**：设备往某个 topic 发送数据
- **Subscribe**：平台或设备订阅某个 topic 来接收数据

不需要深入协议细节，但这几个概念在接 ThingsBoard 时一定会碰到。

总之，这一阶段只要建立下面这个完整链路的概念就够：

- 设备采集数据
- 设备通过某种通信链路（Wi-Fi / 4G / LoRa 等）把数据发出去
- 通过某种协议（HTTP / MQTT）把数据送到平台
- 平台接收并展示数据
- 平台也可以把控制指令发回设备

## 23. 3个月到半年怎么安排

### 23.1 第一阶段：能独立搭环境、跑通基础工程

目标：

- 会安装和配置开发环境
- 会创建和编译工程
- 会烧录和调试
- 会点灯、打印串口、处理中断

### 23.2 第二阶段：掌握常用外设，具备基本调试能力

目标：

- 掌握 UART、I2C、SPI、ADC、DMA
- 遇到基础问题能自己排查
- 开始形成查手册和看波形的习惯

### 23.3 第三阶段：能完成一个小系统

目标：

- 能把多个模块组织起来
- 能完成简单数据流处理
- 能做一个传感器采集或数据上传类小项目

### 23.4 第四阶段：开始具备比赛 / 毕设入门能力

目标：

- 能做基础采样和滤波
- 能做基础人机交互
- 能做基础数据上传
- 能接入平台

## 24. 常见误区

### 24.1 只会抄例程

例程只能帮你起步，不能替代你自己的理解。

### 24.2 只会配置，不会调试

很多人学了一堆外设，真正项目里一出问题就不会查，这会严重影响后续成长。

### 24.3 不查手册

不会查手册，后面做比赛和项目会非常被动。

### 24.4 不做自己的积累

包括：

- 不整理笔记
- 不整理驱动模板
- 不总结常见问题

这些都会让你每次做项目都像重新开始。

### 24.5 一开始目标定得太大

刚入门时更适合从几个有代表性的小系统做起，而不是一上来就做特别复杂的大项目。

## 25. 后续进阶方向

当你把这条路线走完之后，可以继续往这些方向走：

- RTOS（建议从 FreeRTOS 开始，CubeMX 直接集成，上手成本最低）
- 低功耗模式（Sleep / Stop / Standby，物联网方向绕不开）
- Bootloader / OTA（固件升级能力，物联网设备的刚需）
- 驱动抽象与架构设计
- 复杂开源工程阅读
- Zephyr
- RA 系列
- 其他国产兼容 MCU

关于 RTOS 的选择：FreeRTOS 适合从裸机过渡的第一步，生态成熟、资料多；Zephyr 更现代但学习曲线更陡，适合后续进阶。两者定位不同，不冲突。

其中 `RA` 系列可以作为一个扩展方向简单了解，但本文不展开。

## 26. 给新人的最后建议

STM32 的学习不是比谁最早把所有外设都学完，而是比谁更早建立起：

- 工程意识
- 调试意识
- 抽象能力
- 查资料和解决问题的能力

如果你能做到下面这几点，后面成长会快很多：

- 学一个外设，就想它在项目里怎么用
- 写一个驱动，就想它以后怎么复用
- 遇到问题，先看现象、再排查，不要盲改
- 平时把自己的代码、笔记、踩坑整理下来

真正能拉开差距的，往往不是你背了多少库函数，而是你能不能把一个系统从“想法”做成“能跑、能调、能维护”的工程。
